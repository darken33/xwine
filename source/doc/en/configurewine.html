4.1. Configuration Panel

In order to help to configure Wine, Xwine offers a graphical configuration panel. Click on the menu Start->Parameters->Wine Configuration  :


You may execute the following actions :
  • Configure the disk drives
  • Configure the system directories
  • Configure the Windows version to emulate
  • Configure the DLLs
  • Configure the look and feel of the windows
  • Configure the graphical management
  • Configure the ports
  • Configure the registry
  • Configure the terminal
  • Configurer the multimedia drivers
  • Run the WineSetuptk tool (if installed)
  • Edit the Wine config file (for experts only)
  • Configure the applications under Wine
  • Start/Stop the Debug mode
  • Display the Wine log file
  • Restore the default Wine config file
  • Export the Wine configuration / the applications configurations (root only)

4.2. Disk Drives Configuration

This section allows to configure the access to the disk drives under Wine.
From the Configuration Panel, click on the "Drives" button to access the list of drives identified by Wine.

4.2.1. Disk Drives current list



This list displays all disk drives which are identified by Wine.
You may here add, modify or remove a disk drive.

4.2.2. Disk Drives Properties



On that window you may specify the following parameters :
  • the  drive name under Wine (i.e. : [Drive C] for C:)
  • the path that Wine uses to find the drive (i.e. : /mnt/windows) ; you can browse it by clicking on the [...] button
  • the drive type (floppy : floppy disk,  hd : hard disk, cdrom : CDrom drive, network : network drive)
  • the name you give to the drive (i.e. : Windows)
  • the file system type, which describes how are the files managed (win95, msdos, unix)
  • the serial number (optional)
  • the corresponding Linux device (optional), i.e. : /dev/fd0 for the  floppy disk

4.3. System Directories Configuration



You can set here the various environment variables for Wine :
  • Windows : the directory where Windows is installed (the most often c:\\windows)
  • System : the Windows system directory (the most often c:\\windows\\system)
  • Temp : the temporary directory used by Wine
  • Profile : the user profile directory (the most often c:\\windows\\Profiles\\Administrator)
  • Path : the PATH environment variable for Wine (i.e. : c:\\windows;c:\\windows\\system)

4.4. Windows Version Configuration



You may specify the Ms-Windows version to emulate (useful if Windows is installed).
  • Windows version : win95, win98, winxp, winme, win2k (Win 2000), nt40 (NT 4.0), nt351 (NT  3.51) , win20 (Win 2.0), win30 (Win 3.0), win31
  • Ms-Dos 6.22 Emulation : check it if you want to emulate msdos (works only with win31 mode)

4.5. DLLs Configuration

This section allows to parameter the DLL management under Wine.
There are 3 choices for each DLL :
  • native : for use of Ms-Windows DLLs
  • builtin : for use of Wine DLLs
  • so : for use of the Linux system libraries (for the libraries which are identical for Windows and Linux)

4.5.1. DLLs list



This list defines the default order to look for  the  DLLs (native, builtin ou so), and displays the DLLs which have specific loading parameters.
You may also add, modify or remove a DLL.

Below is a configuration which should works for most of the cases (if you have Ms-Windows) :
  • Default : builtin, native, so
  • krnl386, kernel32 : builtin
  • gdi, gdi32 : builtin
  • user, user32 : builtin
  • ntdll : builtin
  • w32skrnl : builtin
  • Win32s (for Win3.x) : builtin.
  • wow32 : builtin
  • system : builtin
  • display : builtin
  • toolhelp : builtin
  • ver, version : native, builtin
  • advapi32 : native, builtin
  • commdlg, comdlg32 : native, builtin
  • commctrl, comctl32 : native, builtin
  • shell, shell32 : native, builtin
  • winsock, wsock32 : builtin
  • icmp : builtin
  • mpr : builtin
  • lzexpand, lz32 : builtin
  • winaspi, wnaspi32 : builtin
  • crtdll : native, builtin
  • winspool.drv : native, builtin
  • mmsystem, winmm : builtin
  • msacm, msacm32 : builtin
  • msvideo, msvfw32 : native
  • mcicda.drv : native, builtin
  • mciseq.drv : native, builtin
  • mciwave.drv : native, builtin
  • mciavi.drv : native, builtin
  • mcianim.drv : native, builtin
  • msacm.drv : builtin
  • midimap.drv : native, builtin
  • wprocs : builtin

4.5.2. Configure a DLL



When you click on New or Modify in the previous list, you get this dialog box , which allows to specify the order to look for a given DLL.

4.6. Look and Feel Configuration



You may choose the look of the windows :
  • Look : choose Win31, Win95 or Win98
  • Fonts : you may choose the size and the type style (Standard, Bold, Italic, Underlined, Strikeout)

4.7. Video Configuration



You will specify here the behavior of the applications windows under Wine :
  • Video Mode : XWindow (managed by the Linux window manager ), .... , Desktop
  • Double Buffer : activates the double buffering, it may be very useful with OpenGL
  • System Colors  : specifies the maximum number of shared colors that  Wine will be able to use. The higher this value is, the lesser the colors will be available for other applications
  • Use DGA :  specifies that you want that DirectDraw use the Direct Graphical  Architecture (DGA) of XFree86, which will manage the display and allow the execution of a game in full screen and at the maximum speed
  • Use XShm : If you do not want that DirectX uses DGA, you may use the Shared Memory Extensions of X (XShm). It is slower than DGA, but faster than the standard display with X11.
  • Use DXGrab : If you do not use DGA, it allows to force the mouse to stay in the game window

4.8. Ports Configuration



You may configure the serial and parallel ports : you may define the corresponding names of a Windows port and a Linux device.
For example : the most often the serial port is named Com1 under Windows and /dev/ttyS0 under Linux.

4.9. Registry Configuration



Allows to specify how Wine manages the Registry :
  • Load the general database from /etc :  implies most often that you have no Ms-Windows
  • Load/Record in home registry : implies that you want to allow the user to modify the Registry (the changes will be stored in his .wine directory  and do not change neither the Windows registry, neither the Wine base)
  • Load Ms-Windows Registry implies that you have Ms-Windows and you want Wine uses its Registry

4.10. Console Configuration



This box allows to define the default Wine terminal. This one is especially started when the debugger is executed.

4.11. Multimedia Configuration



It allows to specify the drivers to use for the sound under Wine :

Drivers : indicates the driver used under Linux (wineoss.drv ou winearts.drv)
Wave : indicates the driver for audio (the most often msacm.drv)
Midi : indicates the driver for midi (the most often midmap.drv)

4.12. Configure with WineSetupTk

Xwine allows to configure Wine with the WineSetupTk tool in a relatively automatical way.
You must have installed WineSetupTk to use this functionality.

4.13. Edit the Wine configuration file



You may here edit the Wine config file, stored in ~/.wine/config.
It allows to configure Wine without using the graphical configuration mode (to be used only if you know exactly what you are doing).
When you click on <OK>, your changes will be stored in the ~/.wine/config file.

4.14. Restore the Wine default configuration file

If you have damaged you configuration file, you may restore constantly the original /etc/wine/config file.

4.15. Export the Wine configuration / applications configurations (root only)



As root, you may export your Wine configuration and the applications configurations, to use them as reference for other users.
  • The Wine configuration file will be exported to /etc/wine/config
  • The applications configurations files will be exported to /etc/xwine/apps/
When starting, XWine verifies if the reference files (/etc/wine/config and the files stored in /etc/xwine/apps/) are more recent than those of the user; if it is true, it loads these ones.

When exporting each application configuration file, XWine asks you to add an icon in the Linux menu corresponding to the application, so that everybody can execute that application without running Xwine (as if it were a Linux application).
